- 首屏为什么慢

## 2. Flutter 基础知识点

### 2.1 Widget、Element、RenderObject 的关系

#### 面试常问
- Widget 和 Element 的区别是什么？
- Flutter 为什么性能高？
- build 过程到底发生了什么？

#### 解答
Flutter 的 UI 是声明式的，核心是三层结构：

- `Widget`：配置描述，轻量、不可变
- `Element`：Widget 在树中的实例，负责挂载、更新、生命周期
- `RenderObject`：真正负责布局、绘制、命中测试

#### 真实开发理解
- `Widget` 变化不等于 UI 重建全部重绘
- Flutter 会通过 `Element` 尽量复用渲染对象
- 频繁重建 Widget 不一定有问题，关键是是否引发了昂贵的布局/绘制

#### 面试答题建议
可强调：
> Flutter 的高性能来自于其组件树和渲染树分离，以及尽量减少原生桥接开销。

---

### 2.2 StatelessWidget 与 StatefulWidget

#### 面试常问
- 什么场景用 StatelessWidget，什么场景用 StatefulWidget？
- setState 做了什么？
- 为什么 setState 不生效？

#### 解答
- `StatelessWidget`：依赖外部输入，自己不持有可变状态
- `StatefulWidget`：内部持有状态，状态变化后通过 `setState` 触发重建

#### 真实开发中的注意点
- 页面状态不一定都放在 `State` 里，很多状态更适合放在 ViewModel/Controller
- `setState` 只适合局部、小范围状态变化
- 如果状态复杂，建议使用状态管理方案

#### 常见坑
- 在异步回调里忘记判断 `mounted`
- 在 `build` 中发起网络请求导致重复请求
- 把所有状态都塞进一个 State 类里，后期难维护

---

### 2.3 BuildContext 的本质

#### 面试常问
- BuildContext 是什么？
- 为什么不能在 initState 里直接依赖某些 context 信息？

#### 解答
`BuildContext` 本质上是 `Element` 的引用，用于定位 Widget 树位置，并访问上层依赖（如 `Theme`、`MediaQuery`、`Navigator`、`Provider`）。

#### 实战建议
- 在 `initState` 中避免直接执行依赖 InheritedWidget 的逻辑
- 若需要 context，可放到 `didChangeDependencies` 或 `addPostFrameCallback`

## 3. 状态管理知识点

### 3.1 为什么状态管理是重点

Flutter 面试中，状态管理几乎是必问项。不是背 API，而是看你能否根据业务复杂度做选型。

### 3.2 常见方案对比

#### setState
适合：
- 局部状态
- 简单交互

优点：简单直接
缺点：复杂页面易失控

#### Provider / ChangeNotifier
适合：
- 中小型项目
- 需要基础共享状态
- 团队想要低学习成本方案

优点：简单、官方推荐方向之一、上手快
缺点：状态分散后可能出现 rebuild 较多、依赖链不够清晰

#### Riverpod
适合：
- 中大型项目
- 复杂依赖管理
- 希望更强的可测试性与解耦

优点：更现代、组合能力强、测试友好
缺点：学习曲线略高

#### Bloc / Cubit
适合：
- 状态变化路径清晰的业务
- 更强调事件流和状态流

优点：结构清晰、可预测
缺点：样板代码偏多

### 3.3 面试回答建议
如果被问“你会选什么状态管理”，不要只答一个名字，要回答选型逻辑：

- 简单页面：`setState`
- 中小项目：`Provider`
- 中大型复杂业务：`Riverpod` 或 `Bloc`
- 团队已有规范：优先沿用团队现有方案

### 3.4 真实开发中的经验点

- 将 UI 状态、网络状态、业务状态分开管理
- 避免页面直接调用接口后到处 setState
- 对于列表页、详情页、登录态建议使用统一 ViewModel / Repository / Service 分层

## 4. 项目架构与分层设计

### 4.1 面试会问什么
- 你的 Flutter 项目怎么分层？
- Repository、ViewModel、Service 有什么区别？
- UI 和数据怎么解耦？

### 4.2 推荐分层思路

典型分层：

- `pages`：负责页面布局和交互
- `viewmodels`：负责状态管理、页面逻辑
- `repositories`：负责业务数据聚合
- `services` / `network`：负责网络访问、存储、第三方能力封装
- `models`：负责实体与序列化
- `core`：通用工具、配置、常量、帮助类

### 4.3 为什么要分层

- 便于维护
- 便于测试
- 降低页面复杂度
- 后续接口变更只改 Repository/Service

### 4.4 面试加分点
你可以结合真实项目说：
> 我会把页面只保留展示和轻交互，复杂业务状态放到 ViewModel，数据来源统一走 Repository，这样页面不会膨胀，也方便替换接口和增加缓存。

## 5. 路由与页面跳转

### 5.1 常问知识点
- Navigator 1.0 和 2.0 的区别
- 什么场景用命名路由
- 如何做登录拦截
- 如何传参和返回结果

### 5.2 实战建议
- 小项目或简单业务：命名路由足够
- 中大型项目：建议统一封装路由入口
- 有深链、Web、复杂嵌套路由时可考虑更完整路由方案

### 5.3 登录拦截
面试里经常会问：
- 401 怎么处理？
- token 失效如何全局跳登录？

建议答法：
1. 网络层识别 401
2. 统一抛出认证异常
3. UI 层或全局拦截器统一跳转登录
4. 清理本地 token 和用户信息

## 6. 网络请求与数据处理

### 6.1 网络层设计

面试常问：
- 你怎么封装网络请求？
- 如何处理错误码？
- 如何处理 token？
- 如何区分业务错误和网络错误？

### 6.2 推荐设计

- `NetworkClient`：统一 http 请求、编码、解码、错误抛出
- `ApiService`：按接口聚合
- `Repository`：做业务接口封装与组合
- `ViewModel`：触发请求并转换为页面状态

### 6.3 错误处理建议

要区分：
- 网络错误：超时、断网、DNS 失败
- 服务端错误：500、502
- 业务错误：code 不正确、字段缺失、权限不足
- 鉴权错误：401、403

### 6.4 面试亮点
可以提到：
- 统一处理 token
- 统一 header
- 统一异常类
- 网络失败时做可恢复 UI
- 列表、详情页做重试机制

## 7. 性能优化知识点

这是 2026 年 Flutter 面试最容易拉开差距的部分。

### 7.1 首屏性能

#### 常问
- 如何优化首屏加载？
- 为什么首屏慢？
- 怎么看首屏耗时？

#### 常见原因
- 首屏请求太多
- 图片过大
- 同步耗时过多
- widget 树太深
- 初始化逻辑过重

#### 优化方法
- 首屏只请求关键数据
- 其他数据延迟加载
- 使用缓存和本地数据兜底
- 减少首屏图片解码压力
- 拆分首屏组件，减少 build 压力

### 7.2 列表性能

#### 常问
- ListView 和 GridView 如何优化？
- 大列表卡顿怎么办？

#### 优化思路
- 使用懒加载构建（builder）
- 控制 item 复杂度
- 使用 `const` 构造
- 合理使用 `RepaintBoundary`
- 图片缩略图/缓存
- 长列表分页加载

#### 真实开发中的坑
- item 里嵌套复杂布局、多个异步组件
- 每个 item 都触发重建
- 图片没有做缓存或尺寸控制
- 不必要的状态更新导致全列表刷新

### 7.3 rebuild 优化

#### 核心原则
- 让需要更新的区域最小化
- 让无关组件不要重建

#### 手段
- 拆分组件
- 使用 `const`
- 控制 Provider/Riverpod 的监听范围
- 避免把整个页面包在大范围状态监听中
- 用 `Selector` / `Consumer` 精准订阅

### 7.4 图片优化

Flutter 中图片是性能问题高发区。

#### 常见问题
- 原图太大导致内存占用高
- 网络波动导致加载卡顿
- 多图同时加载引起掉帧

#### 优化方法
- 使用合适分辨率图片
- 使用缓存图片组件或封装缓存策略
- 占位图 + 渐进式加载
- 列表中避免一次加载太多大图
- 使用 `Image.network` 时控制 `cacheWidth` / `cacheHeight`（按场景）

### 7.5 动画与渲染

#### 常问
- 为什么动画卡？
- 如何优化过渡动画？

#### 建议
- 避免在动画中做重建
- 动画内容尽量轻量
- 避免高频复杂阴影和模糊
- 注意 `BackdropFilter`、大面积渐变、半透明叠层

### 7.6 内存与资源管理

#### 常问
- 页面返回后资源怎么释放？
- 为什么有内存泄漏？

#### 注意点
- `AnimationController` 记得 dispose
- `TextEditingController`、`ScrollController` 记得 dispose
- 流订阅要取消
- 图片缓存要控制
- 长生命周期对象别持有短生命周期 context

## 8. 组件设计与 UI 工程能力

### 8.1 为什么要问组件化
Flutter 项目容易出现“页面越来越大”，所以面试会看你是否具备拆组件能力。

### 8.2 组件化原则
- 单一职责
- 高内聚低耦合
- 可复用、可测试
- 尽量参数化而不是复制粘贴

### 8.3 常见高频场景
- 卡片组件
- 通用按钮
- 空状态、错误状态、加载态
- 自定义 AppBar
- 标签、筛选栏、底部操作栏

### 8.4 面试加分点
如果你能解释：
> 我会把 UI 组件拆成“展示组件”和“业务组件”，展示组件只负责渲染，业务组件负责数据拉取和状态切换。这样复用率更高，也方便调试。

会很加分。

## 10. 数据持久化与本地缓存

### 10.1 常问
- 你用什么存储登录态？
- 怎么缓存列表数据？
- 如何做离线兜底？

### 10.2 常见方案
- `SharedPreferences`：轻量键值存储
- 文件：适合图片、草稿、导出内容
- 数据库：适合复杂结构和列表缓存
- 内存缓存：适合短期状态

### 10.3 实战建议
- token、用户信息使用本地存储
- 列表数据可做简单缓存提升首屏体验
- 缓存要有过期策略
- 网络成功后刷新缓存

## 12. 调试与问题定位能力

2026 年面试很看重“你会不会排查问题”。

### 12.1 常见排查方向
- 网络请求失败：看 URL、header、token、状态码、响应体
- UI 不刷新：看状态是否变化、是否触发监听、是否被缓存
- 卡顿：看 build、layout、paint 哪一步耗时
- 崩溃：看异常栈、空指针、类型转换、异步时序

### 12.2 推荐调试习惯
- 先看日志，再看代码
- 先判断是 UI 问题还是数据问题
- 先缩小问题范围再动代码
- 不要凭感觉改

## 16. 面试回答模板建议

### 16.1 回答结构
建议使用：

- 是什么
- 为什么
- 怎么做
- 真实项目中怎么用

### 16.2 示例：如何回答“你怎么优化列表性能？”
可以这样答：

> 我会先确认卡顿来源，是 build、layout 还是图片加载。然后把列表改成 builder 懒加载，减少 item 内部的复杂布局，图片做缓存和尺寸控制，必要时拆分局部状态，避免整页重建。如果是超长列表，还会配合分页和预加载，并用 DevTools 验证优化效果。

### 16.3 示例：如何回答“你怎么处理 401？”
可以这样答：

> 我会在网络层统一识别 401，抛出专门的未授权异常；在业务层清理 token 和用户信息，然后跳转登录页。这样不会把原始异常直接暴露给用户，也能保证所有页面的鉴权逻辑一致。
