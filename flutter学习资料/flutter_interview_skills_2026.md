# 2026 年 Flutter 面试技能点总结

> 结合真实项目 `oneww` 的实践经验整理，偏向“能讲清楚、能落地、能追问”的面试表达。

## 1. 项目架构能力

### 面试官常问
- 你们项目是什么架构？为什么这么设计？
- 页面、接口、状态、路由怎么组织？
- 如何保证业务扩展时不失控？

### 真实项目总结
在 `oneww` 这种中型 Flutter 项目里，比较合理的做法不是一上来追求过度抽象，而是先按“业务可分层”组织：

- `pages/`：页面层，负责 UI 展示与交互
- `viewmodels/`：状态管理层，负责加载、刷新、状态切换
- `core/network/`：统一网络层，封装请求、拦截、错误处理
- `core/helpers/`：工具、缓存、日志、鉴权等通用能力
- `models/`：数据模型，负责和后端 JSON 对接
- `router/`：统一路由配置和页面守卫

### 面试表达要点
- 先保证“能快速迭代”，再逐步做“结构治理”
- 页面和接口不要耦得太死，推荐用 ViewModel / Service 分离
- 统一封装网络、存储、鉴权、路由，避免业务散落各处

## 2. 路由与统一鉴权

### 面试官常问
- Flutter 路由怎么做？
- 如何控制未登录用户访问受限页面？
- 登录态失效怎么处理？

### 真实项目总结
在 `oneww` 里，登录后访问受限页面时，需要统一鉴权拦截。实践上要做到：

- 路由入口统一走 `onGenerateRoute`
- 受保护页面统一包一层 Guard
- Guard 内部先判断本地 token 是否存在
- 未登录时直接跳转登录页

### 关键点
- 不要在每个页面里单独写一套登录判断
- 路由守卫最好是“统一入口”
- 登录页跳回主页面时，应该清空旧路由栈，避免回退到受限页面

### 面试表达示例
> 我们项目里是通过统一路由入口做鉴权，受限页面统一走 Guard。这样登录态控制集中、可维护性更高，而且后续如果要加 token 刷新、强制下线、权限分级，也方便扩展。

## 3. 登录、退出登录与会话管理

### 面试官常问
- token 存哪里？
- 退出登录怎么处理？
- 除了 token，还会保存什么信息？

### 真实项目总结
`oneww` 中登录态不只保存 token，还会保存 `userInfo`。

退出登录时要做到：

- 清除 token
- 清除 userInfo
- 回到登录页
- 清空路由栈

### 经验点
- token 存储通常用 `SharedPreferences` 或安全存储
- 如果后端返回 userInfo，建议登录时一并缓存，减少首页或个人中心的重复请求
- 退出时一定要“一次性清理干净”，避免残留状态导致 UI 误判

### 面试表达示例
> 我们把 token 和 userInfo 统一封装在 AuthStorage 里，登录成功后同步保存，退出登录时统一清空。这样个人中心、首页欢迎信息、鉴权逻辑都可以直接复用，不需要到处查接口。

## 4. 网络请求封装与接口设计

### 面试官常问
- 你们项目怎么封装网络层？
- 如何处理请求头、token、错误和日志？
- 怎么应对不同后端接口返回格式？

### 真实项目总结
在 `oneww` 中，网络层建议按“统一入口 + 业务接口封装”来做：

- `NetworkClient` 负责底层请求
- `ApiService` 负责业务接口方法
- `NetworkEndpoints` 管理接口地址常量
- `AppLogger` 记录调试信息和错误日志

### 实践经验
- token 不要在页面里手动拼接
- 请求头统一处理，避免不同页面写不同规则
- 对接口返回值做模型映射，别让 UI 直接解析 JSON
- 接口地址常量集中管理，减少手写错误

### 面试表达示例
> 我们会把鉴权头、设备标识、错误日志统一放在网络层处理。业务接口只暴露方法和模型，页面层只关心拿到什么数据，不关心怎么发请求。

## 5. 数据模型与 JSON 解析

### 面试官常问
- 如何把后端 JSON 转成 Dart 对象？
- 模型字段和接口字段不一致怎么办？
- 嵌套列表怎么解析？

### 真实项目总结
在业务项目里，模型层非常重要。比如首页里：

- 分类列表是一个模型
- 推荐列表又是一个模型
- 登录会话也可以抽成单独模型

### 技术要点
- 使用 `factory fromJson(Map<String, dynamic>)`
- 尽量给默认值，减少空值带来的 UI 崩溃
- 嵌套对象、列表对象要逐层拆解
- JSON 字段命名和 Dart 命名不一致时，要在模型层统一适配

### 面试表达示例
> 我们会优先把 JSON 解析集中在模型层，不让 UI 做字符串判断和字段兜底。这样后端字段变动时，只需要改模型映射，不会影响整个页面。

## 6. 状态管理与页面刷新

### 面试官常问
- Flutter 页面状态怎么管理？
- 切换分类后怎么刷新列表？
- 如何避免重复请求和状态串台？

### 真实项目总结
在首页这种复杂页面里，状态至少分成三类：

- 分类加载状态
- 推荐列表加载状态
- 当前选中分类状态

实践中要注意：

- 分类切换时重置分页
- 防止重复点击导致多个请求并发
- 避免旧请求返回覆盖新请求结果

### 经验点
- 请求中要有 loading 标记
- 切换分类前要判断是否真的切换了
- 分页接口应支持追加而不是整页覆盖

### 面试表达示例
> 我们首页分类切换会带请求防抖和请求去重，避免重复点击导致接口风暴。同时列表分页和分类状态是绑定的，切换分类会重置第一页，保证展示数据一致。

## 7. 分页加载与性能优化

### 面试官常问
- Flutter 列表怎么做分页？
- 如何避免列表卡顿？
- 大量数据如何处理？

### 真实项目总结
推荐列表这类内容很适合分页加载：

- 首次进入加载第一页
- 滚动到底继续请求下一页
- 新数据追加到列表尾部
- 适配空态、加载中、错误态

### 常见优化点
- 使用惰性构建列表
- 图片做缓存和尺寸控制
- 控制卡片高度，避免布局抖动
- 复杂列表建议分组件拆分，减少 build 压力

### 面试表达示例
> 我们会把列表做成分页加载，配合加载中骨架、空态和错误态，既保证体验，也能避免一次性拉太多数据造成卡顿。

## 8. UI 还原与像素级调优

### 面试官常问
- 你怎么做 UI 还原？
- 怎么让页面看起来更像设计稿？
- 如何减少布局溢出？

### 真实项目总结
Flutter 面试里，UI 还原不是单纯“写得像”，而是要体现：

- 层级清晰
- 间距一致
- 组件复用
- 溢出控制

### 实战经验
在首页这种场景里，常见做法是：

- 顶部用 `SafeArea` 防止状态栏遮挡
- 使用 `CustomScrollView + Sliver` 提升可控性
- 固定卡片高度，降低 overflow 风险
- 用统一的圆角、阴影、颜色体系维持风格一致

### 面试表达示例
> 我们会先搭页面骨架，再逐步微调间距、字体、圆角和阴影。对于复杂首页，会尽量用 Sliver 和固定高度卡片，减少底部 overflow 和重绘问题。

## 9. 调试、日志与问题定位

### 面试官常问
- 遇到接口问题怎么排查？
- 页面报红、布局溢出怎么定位？
- 你平时怎么做日志？

### 真实项目总结
在真实项目里，日志非常关键，尤其是：

- 登录 token 是否带上了
- 请求参数是否正确
- 返回体结构是否符合预期
- 页面溢出是否由高度变化触发

### 建议
- 网络层记录请求 URL、请求头和关键错误
- 页面逻辑中记录关键状态切换点
- 不要在 UI 里滥用 `print`，统一日志工具更可控

## 10. 面试中可以主动讲的“真实项目亮点”

你可以围绕这些内容展开：

1. **统一鉴权**
   - 登录状态由路由守卫控制
   - 退出登录能清理 token 和 userInfo

2. **会话管理**
   - 登录成功后保存 token + userInfo
   - 个人中心能直接读取本地用户信息

3. **动态首页**
   - 分类由接口驱动
   - 点击分类可刷新推荐列表
   - 支持分页加载、空态、错误态

4. **网络封装**
   - API 统一集中管理
   - 请求头统一注入
   - 返回结果统一映射模型

5. **UI 体验**
   - 首页按设计稿做了多轮精修
   - 重点处理了 overflow 和布局稳定性

## 11. 面试时的回答模板

### 模板 1：说架构
> 我们项目整体是按页面、状态、网络、模型分层的。页面只负责展示和交互，接口和缓存逻辑放在统一 service 和 helper 里，这样后续扩展新业务时不会把逻辑散在页面中。

### 模板 2：说登录态
> 登录成功后会保存 token 和 userInfo，退出登录时统一清空。路由层也有统一鉴权，避免未登录用户直接进入受限页面。

### 模板 3：说列表分页
> 推荐列表采用分页加载，切换分类会重置页码并重新请求，同时做了请求去重，避免重复点击带来的并发问题。

### 模板 4：说 UI 还原
> UI 还原不是只看像不像，还要看布局是否稳定、交互是否顺滑、是否能适配不同屏幕。我们会用固定高度、Sliver 和统一视觉规范来减少 overflow 和抖动。
