### 5.1 弱网场景音视频通信优化

```
问题描述：
在单兵战术场景下，网络环境复杂（地下室、山区、电磁干扰），
需要保证音视频通信的稳定性和可用性。

解决方案：
┌─────────────────────────────────────────────────────────────────┐
│                      弱网优化体系                                 │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│  1. 带宽估计与自适应                                               │
│     ├── GCC（Google Congestion Control）                        │
│     ├── 动态码率调整                                              │
│     └── 降级策略（帧率/分辨率/编码质量）                          │
│                                                                 │
│  2. 抗丢包机制                                                    │
│     ├── NACK + RTX（重传）                                       │
│     ├── FEC（前向纠错）                                           │
│     └── PLC（丢包补偿）                                          │
│                                                                 │
│  3. 缓冲策略                                                     │
│     ├── Jitter Buffer动态调整                                    │
│     ├── 自适应延迟                                               │
│     └── 抗抖动能力                                               │
│                                                                 │
│  4. 音频优先策略                                                 │
│     ├── 音频保真                                                 │
│     ├── 静音检测（VAD）                                          │
│     └── 带宽分配优先级                                           │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘

技术细节：
├── 带宽估计：基于RTCP RR报文计算RTT和丢包率
├── 码率调整：根据带宽估计动态调整编码器目标码率
├── 重传策略：NACK触发后等待1-2个RTT再重传
├── FEC冗余：根据丢包率动态调整FEC冗余度

关键指标：
├── 抗丢包能力：30%丢包下保持通话
├── 端到端延迟：<300ms
└── 恢复时间：网络恢复后<2s恢复正常质量
```

### 10.1 模块化架构设计追问

#### 模块间通信如何实现？

**参考答案：**

模块间通信主要有以下几种实现方式：

**1. 依赖注入（Dependency Injection）**
```kotlin
// 通过接口抽象模块依赖
interface IMapModule {
    fun getMapController(): MapController
    fun setMapClickListener(listener: OnMapClickListener)
}

// 模块提供方实现
class MapModuleImpl : IMapModule {
    override fun getMapController(): MapController = mapController
}

// 使用方注入
class PlotModule(private val mapModule: IMapModule) {
    fun syncWithMap() {
        val controller = mapModule.getMapController()
        // 进行标绘同步
    }
}
```

**2. 事件总线（EventBus）**
```kotlin
// 使用EventBus进行模块间解耦通信
@Subscribe(threadMode = ThreadMode.MAIN)
fun onPlotChanged(event: PlotChangedEvent) {
    // 接收标绘变化事件
}

// 发布事件
EventBus.getDefault().post(PlotChangedEvent(plotId, newPosition))
```

**3. 路由跳转（ARouter/Uri Scheme）**
```kotlin
// 声明路由
@Route(path = "/plot/detail")
class PlotDetailActivity : Activity()

// 跳转
ARouter.getInstance()
    .build("/plot/detail")
    .withLong("plotId", plotId)
    .navigation()
```

**4. HarmonyOS Module通信**
```typescript
// 使用AppStartup进行模块初始化依赖管理
// 使用EventHub进行事件发布订阅
import common.EventHub from '@ohos.common.EventHub';

this.context.eventHub.on('plotChanged', (data) => {
    // 处理标绘变化
});
```

**面试加分点：**
- 说明你实际项目中使用的方案
- 解释为什么选择该方案（解耦度、复杂度、性能权衡）
- 提及潜在的循环依赖问题和解决方式

```kotlin
object MercatorProjection {
    private const val EARTH_RADIUS = 6378137.0
    private const val MAX_LATITUDE = 85.051128779806604
    
    fun lonLatToPixel(lon: Double, lat: Double, zoom: Int, tileSize: Int = 256): Point {
        val latClamped = lat.coerceIn(-MAX_LATITUDE, MAX_LATITUDE)
        
        val x = (lon + 180.0) / 360.0 * tileSize * (1 shl zoom)
        val y = (1.0 - ln(tan(Math.toRadians(latClamped)) + 1.0 / cos(Math.toRadians(latClamped)))

        return Point(x, y)
    }
```

```kotlin
private fun catmullRom(p0: GeoPoint, p1: GeoPoint, p2: GeoPoint, p3: GeoPoint, t: Float): GeoPoint {
```

#### FEC与NACK区别和使用场景？

**参考答案：**

| 机制 | 全称 | 原理 | 适用场景 | 代价 |
|------|------|------|----------|------|
| **FEC** | Forward Error Correction | 发送端添加冗余数据，接收端直接恢复 | 高延迟容忍、连续丢包 | 增加带宽 |
| **NACK** | Negative Acknowledgment | 接收端请求重传丢失数据 | 低延迟要求、偶尔丢包 | 增加RTT |

**1. FEC前向纠错**
```
发送端：
[包1][包2][包3] + [FEC包=XOR(1,2,3)]

接收端（丢失包2）：
[包1][丢失][包3][FEC]
计算：包1 XOR 包3 XOR FEC = 包2  → 成功恢复！
```

**2. NACK重传机制**

```kotlin
class NackHandler {
    private val nackHistory = mutableMapOf<Int, Long>()
    
    fun onPacketReceived(packet: RtpPacket) {
        nackHistory[packet.sequenceNumber] = System.currentTimeMillis()
        
        val expectedSeq = (lastSeq + 1) % 65536
        if (packet.sequenceNumber != expectedSeq) {
            val lostSeqs = (expectedSeq until packet.sequenceNumber).toList()
            sendNack(lostSeqs)
        }
        lastSeq = packet.sequenceNumber
    }
}
```

**3. 混合策略**

```kotlin

        networkState.rtt < 100 -> NackStrategy(maxRetransmit = 3)
```

#### Jitter Buffer原理？

**参考答案：**

**1. Jitter Buffer作用**

```
网络抖动：
发送:  [包1][包2][包3][包4][包5]  时间均匀
接收:  [包1][  ][包3][包4][包5]  到达不均匀（抖动）

Jitter Buffer缓冲后：
播放:  [包1][包2][包3][包4][包5]  时间均匀

代价：增加延迟
收益：播放流畅
```

**2. 自适应Jitter Buffer实现**

```kotlin
class AdaptiveJitterBuffer {
    private var bufferMs: Int = 40  // 初始缓冲
    private val packets = ArrayDeque<RtpPacket>()
    
    fun adjustBufferSize(arrivalInterPacketMs: Int) {
        val jitter = abs(arrivalInterPacketMs - calculateInterFrameTime())
        
        bufferMs = when {
            jitter > bufferMs * 0.5 -> min(bufferMs + 10, 200)
            jitter < bufferMs * 0.3 && bufferMs > 40 -> max(bufferMs - 5, 40)
            else -> bufferMs
        }
    }
    
    fun getPacket(): RtpPacket? {
        val targetTime = clock.nowMs() - bufferMs
        return packets.find { it.timestamp <= targetTime }?.also { 
            packets.remove(it) 
        }
    }
}
```

| 4G网络 | 60-100ms |
| WiFi弱网 | 80-150ms |
| 极端弱网 | 150-200ms |

```kotlin
object MosCalculator {
    
    fun calculateMOS(rtt: Int, jitter: Int, packetLoss: Float): Double {
        // R因子计算
        var r = 93.2
        r -= 0.024 * rtt + 0.11 * (rtt - 177.3).coerceAtLeast(0.0)
        r -= 0.02 * jitter
        r -= 2.5 * 30 * packetLoss.coerceAtMost(0.05f)
        
        // 转换为MOS
        return when {
            r < 0 -> 1.0
            r > 100 -> 4.5
            else -> 1.0 + 0.035 * r + r * (r - 60) * (100 - r) * 7e-6
        }.coerceIn(1.0, 5.0)
    }
}

/*
MOS评分标准：
5.0 - 最佳（无法察觉）
4.0 - 优秀（有察觉但不烦人）
3.5 - 良好（稍微察觉）
3.0 - 一般（有察觉，有影响）
2.0 - 差（明显察觉，难以忍受）
1.0 - 很差（无法忍受）
*/
```

### 10.7 WebSocket信令交互追问

#### WebSocket与HTTP区别？

**参考答案：**

| 特性 | HTTP | WebSocket |
|------|------|-----------|
| **连接方式** | 请求-响应 | 双向持久 |
| **连接次数** | 每次新建 | 一次握手，长期保持 |
| **头部开销** | 每个请求都带Header | 仅建立时带Header |
| **服务器推送** | 需要轮询 | 原生支持 |
| **适用场景** | REST API | 实时通信 |

**连接生命周期对比**

```
HTTP:
Client ──────── Request ─────────▶ Server
        ◀────── Response ────────
        (每次都需要重新连接)

WebSocket:
Client ──────── HTTP Upgrade ────▶ Server
        ◀────── 101 Switching ─────
              ┌──────────────────┐
              │ 双向帧传输       │
Client ◀──────│ 持久连接        │────▶ Server
              └──────────────────┘
```

#### 心跳机制如何设计？

**参考答案：**

**心跳保活原理**

```
Client ──────────── Ping ─────────────────▶ Server
                  (定期发送，检测连接状态)

Client ◀─────────── Pong ◀──────────────── Server
                  (响应心跳)

├── 尝试重连
```

**完整心跳实现**

```kotlin
class HeartbeatManager {
    private var heartbeatInterval = 30_000L
    private var maxRetryCount = 3
    private var retryCount = 0
    
    suspend fun sendHeartbeat(): Boolean {
        return try {
            val sent = webSocket?.send("heartbeat") ?: false
            if (sent) {
                retryCount = 0
                heartbeatInterval = 30_000L
            }
            sent
        } catch (e: Exception) {
            false
        }
    }
    
    private suspend fun handleHeartbeatFailure() {
        retryCount++
        if (retryCount <= maxRetryCount) {
            heartbeatInterval = minOf(heartbeatInterval / 2, 5000)
        } else {
            onConnectionLost()
        }
    }
}
```

#### 如何处理断线重连？

**参考答案：**

**指数退避重连**

```kotlin
class ReconnectionManager {
    private val initialDelay = 1000L      // 初始1秒
    private val maxDelay = 30_000L        // 最大30秒
    private val multiplier = 2.0          // 指数倍增
    private var currentDelay = initialDelay
    private var attemptCount = 0
    
    fun reconnect() {
        scope.launch {
            while (shouldReconnect()) {
                attemptCount++
                try {
                    if (connect()) {
                        onReconnectSuccess()
                        return@launch
                    }
                } catch (e: Exception) {
                    Log.e("Reconnect", "重连失败", e)
                }
                
                delay(currentDelay)
                currentDelay = minOf(currentDelay * multiplier, maxDelay)
            }
            onReconnectFailed()
        }
    }
    
    private fun shouldReconnect(): Boolean {
        return attemptCount < maxReconnectAttempts && isNetworkAvailable()
    }
}
```
