# 3. Android 技能面试题与答案

## 3.1 Kotlin 基础与进阶

### Q1：Kotlin 相比 Java 的核心优势是什么？
**答：**
Kotlin 的优势主要体现在：

- 语法更简洁，减少样板代码
- 空安全机制减少 NPE 风险
- 协程支持更自然的异步编程
- 扩展函数、高阶函数提高可维护性
- 数据类、密封类、委托、内联函数等特性更适合现代 Android 开发

**分析过程：**
如果只是回答“更简洁”，面试官通常不满意。更好的回答要体现：

1. **开发效率**：少写大量 getter/setter 和回调样板代码；
2. **安全性**：空安全、类型推断减少运行时错误；
3. **架构适配**：协程 + Flow 更适合状态驱动 UI 和响应式开发；
4. **团队协作**：代码意图更清晰，易读性更高。

---

### Q2：高阶函数、扩展函数、Lambda 常用在什么场景？
**答：**
- 高阶函数：用于抽象行为，例如回调、策略模式、事件处理
- 扩展函数：为现有类增加能力，减少工具类臃肿
- Lambda：简化接口实现，配合集合操作、异步回调、Compose 事件绑定

**分析过程：**
在项目中，这些特性通常用于：

- Compose 页面事件处理
- 统一的错误处理封装
- 列表数据变换
- 网络回调封装
- 通信链路状态流转

如果被追问“实际怎么用”，可以答：
> 比如在投屏或音视频场景里，我会用高阶函数抽象信令回调，用扩展函数封装状态转换和数据格式处理，让业务层更清晰。

---

### Q3：data class、sealed class、object 的区别？
**答：**
- `data class`：用于承载数据，自动生成 `equals/hashCode/copy/toString`
- `sealed class`：用于表达有限状态集合，常用于 UI 状态、事件结果、网络返回状态
- `object`：单例对象，适合工具类、常量、状态管理中心

**分析过程：**
在现代 Android 架构中，`sealed class` 很常用于：

- Loading / Success / Error
- 页面状态流转
- 设备连接状态
- 音视频会话状态

比如：
```kotlin
sealed class UiState {
    object Loading : UiState()
    data class Success(val data: List<String>) : UiState()
    data class Error(val msg: String) : UiState()
}
```

## 3.2 协程、Flow、StateFlow

### Q4：为什么 Android 现在更推荐协程而不是传统线程？
**答：**
协程的优势在于：

- 更轻量，不像线程那样占用大量资源
- 支持结构化并发，生命周期更清晰
- 更容易处理异步任务与异常传播
- 与 Flow、Compose、ViewModel 集成更自然

**分析过程：**
传统线程适合底层并发控制，但在 Android 业务层，协程更适合：

- 网络请求
- 设备发现
- 信令交互
- 数据加载
- IO 操作
- 状态同步

它的核心价值不是“替代线程”，而是让异步代码更容易读、写、取消和管理。

---

### Q5：Flow 和 StateFlow 有什么区别？
**答：**
- `Flow`：冷流，只有被收集时才开始执行，适合一次性异步数据流
- `StateFlow`：热流，始终持有最新状态，适合页面状态管理

**分析过程：**
你在项目里可以这样回答：

- 网络请求结果、设备扫描结果、投屏状态变化等适合 Flow
- UI 页面状态、连接状态、开关状态、设置状态适合 StateFlow

Compose 和 StateFlow 配合特别自然，因为 UI 可以直接跟随状态变化重组。

---

### Q6：结构化并发是什么？
**答：**
结构化并发强调协程任务的生命周期要受到父作用域管理，避免任务泄漏、异常失控和资源浪费。

**分析过程：**
在项目里常见场景：

- 页面销毁时取消设备扫描
- 音视频页面退出时关闭采集和传输协程
- 投屏连接中断时取消心跳和状态同步任务

这样可以提升稳定性，避免后台任务残留。

---

## 3.3 Jetpack Compose

### Q7：Compose 相比 XML 的核心价值是什么？
**答：**
Compose 是声明式 UI 框架，核心价值在于：

- UI 与状态绑定
- 减少手工操作 View 的复杂性
- 组件复用更方便
- 状态变更自动触发重组
- 更适合复杂交互和多状态页面

**分析过程：**
对于你这种项目，Compose 很适合：

- 设备列表
- 状态控制面板
- 图层管理面板
- 投屏参数设置页
- Launcher 首页
- 智能音箱设置页

因为这些页面状态复杂、交互频繁，Compose 的状态驱动模型优势明显。

---

### Q8：Compose 中什么是重组？如何避免无意义重组？
**答：**
重组是当依赖状态变化时，Compose 重新执行相关 Composable 函数来更新 UI 的过程。

避免无意义重组的方法：

- 提升状态粒度，避免大对象频繁变更
- 使用 `remember` 缓存局部状态
- 用 `derivedStateOf` 计算派生状态
- 拆分大 Composable，降低重组范围
- 避免在 Composable 中做耗时操作

**分析过程：**
如果被追问，你可以举例：
> 在投屏控制面板里，设备状态、画面状态、码率状态是独立变化的，我会把它们拆开管理，减少整个页面不必要的重组。

---

### Q9：Compose 中 state hoisting 为什么重要？
**答：**
State hoisting 是把状态上移，让 UI 组件尽量成为纯展示组件。这样做有助于：

- 降低耦合
- 提升复用性
- 更容易测试
- 更容易管理复杂页面状态

**分析过程：**
在你的项目中，比如“标绘面板”“投屏设置”“Launcher 卡片”都适合做状态上提，让业务逻辑集中在 ViewModel 或状态容器中。

---

## 3.4 Android 架构

### Q10：MVVM 和 MVI 的区别？
**答：**
- MVVM 更强调 View 与 ViewModel 分离，适合多数业务页面
- MVI 更强调单向数据流和不可变状态，适合复杂交互、状态多、事件多的页面

**分析过程：**
你这种项目，尤其是投屏、地图、音视频、系统设置等高状态复杂度页面，MVI / UDF 的表达会更有优势，因为：

- 状态来源清晰
- 事件流转清晰
- 调试更方便
- 适合 Compose

如果面试官问你为什么不是纯 MVVM，可以回答：
> 我会根据复杂度选择，状态较简单的页面用 MVVM，状态流转复杂、交互多的页面更偏向 MVI / UDF。

---

### Q11：模块化和组件化有什么区别？
**答：**
- 模块化偏向工程拆分，通常按业务或功能拆分 Gradle Module
- 组件化偏向业务能力沉淀，把可复用能力抽象成组件或服务

**分析过程：**
在你的项目里：

- 地图、标绘、演播、导航、量算可以做模块化拆分
- 标绘面板、图层面板、设备控制面板可以做组件化抽象

模块化解决“工程组织”问题，组件化解决“能力复用”问题。

---

## 3.5 网络、通信、音视频

### Q12：WebRTC 的核心组成是什么？
**答：**
WebRTC 核心包括：

- 音视频采集
- 编码/解码
- 信令交互
- ICE 候选收集
- NAT 穿透
- PeerConnection
- 音视频传输
- 渲染播放

**分析过程：**
你在项目里可以把 WebRTC 分成几个层：

1. **信令层**：负责设备配对、会话建立、能力协商；
2. **媒体层**：负责采集、编解码、传输；
3. **渲染层**：负责显示和投屏展示；
4. **优化层**：负责弱网、延迟、花屏、同步问题。

这样回答会更专业。

---

### Q13：WebSocket 在投屏项目里通常怎么用？
**答：**
WebSocket 通常用于信令通信，而不是音视频数据本身。它常用于：

- 设备发现
- 会话邀请
- 设备配对
- 协商参数传输
- 状态同步
- 控制指令下发

**分析过程：**
音视频数据走 WebRTC，控制信令走 WebSocket，这是典型分层方式。这样做能把“控制”和“媒体”分开，便于扩展和调优。

---

### Q14：弱网下怎么优化投屏或视频通话？
**答：**
常见思路包括：

- 自适应码率
- 降低分辨率 / 帧率
- 调整缓冲策略
- 优化关键帧策略
- 调整丢包重传机制
- 选择更合适的编码方案
- 减少不必要的 UI 资源占用

**分析过程：**
弱网优化的本质是“质量、延迟、稳定性”三者平衡。你可以根据场景回答：

- 会议场景：优先稳定与音频清晰
- 游戏场景：优先低延迟
- 影视场景：优先画质
- 家庭场景：优先连接稳定和易用性

---

### Q15：H264、H265、VP8、VP9 怎么理解？
**答：**
它们都是视频编码格式，不同编码在压缩率、兼容性、解码开销上有差异。

- H264：兼容性广，通用性强
- H265：压缩率更高，但计算复杂度更高
- VP8：WebRTC 常见编码，兼容性不错
- VP9：压缩效率更好，但资源开销更高

**分析过程：**
回答时不要死背参数，而要结合场景：

- 端侧性能一般、兼容性要求高：倾向 H264
- 带宽紧张、设备支持好：可考虑 H265
- WebRTC 场景：常见 VP8 / H264 组合

## 3.6 Android 系统与终端

### Q16：Binder / AIDL 常用于什么场景？
**答：**
Binder 是 Android 的 IPC 机制，AIDL 是对 Binder 的接口描述工具。通常用于：

- App 和系统服务之间通信
- 多进程模块交互
- 设备控制
- 系统设置调用
- 智能终端中间层接口封装

**分析过程：**
你在智能音箱项目中可以这样说：
> 我们通过 AIDL 封装系统设置、设备信息、网络蓝牙配置等能力，让 App 层与系统服务层解耦，同时保证跨进程通信稳定。

---

### Q17：Launcher 开发为什么要重视启动优化？
**答：**
Launcher 是系统入口，用户对它的启动速度和稳定性最敏感。优化重点包括：

- 冷启动优化
- 首屏时间优化
- 资源预加载
- 减少主线程阻塞
- 内存优化
- ANR 治理

**分析过程：**
Launcher 的优化不是单纯缩短启动时间，而是要保障“系统入口体验”。如果 Launcher 卡顿，用户会直接感知到系统慢。

---

### Q18：Android 系统签名有什么意义？
**答：**
系统签名用于让应用获得系统级权限，通常用于：

- 系统设置
- 系统服务访问
- Launcher
- 设备管理
- 框架交互

**分析过程：**
智能音箱、TV、车机等项目中，系统级应用常需要更高权限，这时系统签名和权限管理就是重要环节。

# 4. HarmonyOS 技能面试题与答案

## 4.1 ArkTS / ArkUI

### Q19：ArkTS 相比普通 TypeScript/JS 的特点是什么？
**答：**
ArkTS 是 HarmonyOS 面向应用开发的语言，强调：

- 更强的类型约束
- 声明式 UI 友好
- 面向 ArkUI 生态
- 更适合构建鸿蒙应用

**分析过程：**
你可以强调：
> ArkTS 本质上是鸿蒙应用开发体系的一部分，它不是单纯的脚本语言，而是配合 ArkUI、Stage 模型和系统能力来完成整套应用开发。

---

### Q20：ArkUI 声明式开发有什么优势？
**答：**
- 状态与 UI 绑定
- 代码更清晰
- 组件复用更强
- 更适合复杂交互页面
- 更容易进行状态驱动管理

**分析过程：**
你在单兵战术终端中可以把标绘面板、时间轴演播、图层面板等界面描述成典型的状态驱动 UI，非常适配 ArkUI。

---

### Q21：State Management 在鸿蒙项目里怎么理解？
**答：**
状态管理就是将页面状态、组件状态、全局状态进行合理拆分，并通过状态驱动 UI 更新。

**分析过程：**
常见问题是状态过度集中，导致页面臃肿。你可以回答：
> 我通常会将局部交互状态、页面状态和全局状态分层管理，避免多个模块抢状态，提升页面可维护性。

---

## 4.2 Stage 模型

### Q22：Stage 模型的核心是什么？
**答：**
Stage 模型是 HarmonyOS 应用运行和组织的基础模型，核心围绕：

- UIAbility
- ExtensionAbility
- 组件化开发
- 应用生命周期
- 任务栈和页面组织

**分析过程：**
它的重点不是“启动一个页面”，而是“以应用为中心组织多能力协作”。你在简历里写过主 Entry + 多 Module 的架构，这和 Stage 模型工程化很契合。

---

### Q23：UIAbility 和 ExtensionAbility 的区别？
**答：**
- UIAbility：承载用户可见界面和交互
- ExtensionAbility：承载扩展能力，如服务、数据、后台、分享等

**分析过程：**
面试时可以补充：
> UIAbility 更像入口能力，ExtensionAbility 更像系统扩展能力。复杂项目里，我们通常会把 UI 与扩展能力分离，提升架构清晰度。

---

## 4.3 线程、并发、分布式

### Q24：TaskPool 和 Worker 怎么理解？
**答：**
- TaskPool：适合调度异步任务
- Worker：更适合并行计算和后台处理

**分析过程：**
你可以回答：
> 在地图渲染、标绘运算、离线数据处理这类任务中，我会把耗时逻辑放到后台线程，避免阻塞主线程，提升页面流畅性。

---

### Q25：HarmonyOS 分布式数据管理有什么价值？
**答：**
它能够帮助多设备之间实现数据共享、状态同步和协同使用，适合跨设备业务。

**分析过程：**
你可以把它和你的项目结合：
- 单兵终端与指挥设备之间的状态同步
- 标绘数据的同步
- 任务下发与回传
- 多设备协同场景

---

### Q26：Distributed Soft Bus 的作用是什么？
**答：**
Distributed Soft Bus 是鸿蒙设备之间通信和协同的重要基础能力，支持设备发现、连接和通信。

**分析过程：**
如果被追问应用场景，可以回答：
> 多设备协同、动态能力加载、跨设备任务传递、状态同步，都是软总线能力适合支撑的方向。

---

# 5. 音视频 / 投屏 / WebRTC 专项

## 5.1 WebRTC 基础

### Q27：WebRTC 做音视频通话时，最关键的链路是什么？
**答：**
最关键的是：

- 采集
- 编码
- 信令协商
- 网络传输
- 解码
- 渲染

**分析过程：**
每个环节都可能影响最终体验。面试时最好能说出你优化过哪一段，比如：
- 采集端优化降噪和回声消除
- 传输端优化延迟和抗抖动
- 渲染端优化丢帧和卡顿

---

### Q28：WebRTC 信令为什么不能直接用来传音视频数据？
**答：**
信令主要负责建立连接、协商参数和状态同步，不适合直接承载大规模媒体数据。音视频数据应通过更适合实时传输的媒体链路传输。

**分析过程：**
在投屏或视频通话场景中，控制平面和媒体平面分离是标准做法。这样便于扩展、优化和排障。

---

## 5.2 低延迟和弱网优化

### Q29：你会怎么优化低延迟投屏？
**答：**
- 选择适合场景的编码方案
- 降低编码帧率或分辨率
- 调整缓冲和抖动策略
- 减少重传和关键帧开销
- 优化首帧和连接建立耗时
- 采集、编码、传输、渲染全链路排查

**分析过程：**
低延迟不只是“网络快”，而是全链路问题。面试官如果问得深，你可以按“采集—编码—传输—渲染”逐层回答。

---

### Q30：如何定位花屏、卡顿、延迟高？
**答：**
- 先判断问题出在采集、编码、传输还是渲染
- 查看日志和状态统计
- 使用抓包分析传输链路
- 查看协议协商和媒体参数
- 根据场景调节编码、缓冲和渲染策略

**分析过程：**
你可以回答为一个“定位闭环”：
1. 复现问题；
2. 判断问题归属链路；
3. 用日志/统计信息/抓包分析；
4. 针对性优化；
5. 回归验证。

---

### Q31：投屏场景为什么要做差异化参数策略？
**答：**
因为不同场景的目标不同：

- 会议：稳定、音频优先
- 游戏：低延迟优先
- 影视：清晰度优先
- 家庭：易用性和稳定性优先

**分析过程：**
同样是投屏，不同场景的最优解不同。一个成熟工程师会做“场景化策略”，而不是一个参数跑到底。

---

# 6. GIS 地图 / 标绘 / 演播专项

## 6.1 地图与图层

### Q32：你做过的地图模块，难点是什么？
**答：**
主要难点通常在：

- 多图层管理
- 复杂态势渲染
- 坐标转换
- 大数据量加载
- 交互响应
- 离线缓存

**分析过程：**
地图项目不是单纯显示地图，而是要处理“地图 + 业务状态 + 交互 + 性能”的综合问题。你要体现出自己能把地图当作一个业务平台，而不是控件。

---

### Q33：图层管理怎么做更合理？
**答：**
- 图层分级管理
- 图层显示/隐藏状态独立控制
- 图层数据和渲染逻辑分离
- 重要图层支持优先级和联动
- 通过统一管理器控制图层生命周期

**分析过程：**
对于战术地图，图层往往很多，合理的图层体系会直接影响性能和可维护性。

---

## 6.2 标绘与演播

### Q34：军标标绘最难的点是什么？
**答：**
- 标绘元素类型多
- 绘制、编辑、拖拽、组合等交互复杂
- 状态同步和协同难
- 与地图坐标、比例、缩放联动复杂

**分析过程：**
标绘本质上是“几何计算 + UI 交互 + 数据持久化 + 协同同步”的综合问题。你可以说明自己如何做数据模型设计和交互抽象。

---

### Q35：动态演播模块如何设计？
**答：**
动态演播通常需要：

- 时间轴控制
- 场景状态回放
- 暂停/快进/回放
- 与地图状态联动
- 与标绘状态联动

**分析过程：**
动态演播的关键是把业务状态抽象成可回放、可控制、可同步的状态序列，而不是单纯做动画。

---

# 7. 系统级终端 / 智能音箱 / Launcher 专项

## 7.1 系统级开发

### Q36：智能音箱 Launcher 开发最关键的点是什么？
**答：**
- 启动快
- 稳定
- 交互简单
- 支持系统能力入口
- 适配语音、按键、触屏

**分析过程：**
Launcher 是系统入口，首屏体验和稳定性非常重要。你可以强调自己做过启动优化、内存优化和 ANR 治理。

---

### Q37：AIDL 为什么适合系统级终端？
**答：**
AIDL 适合把系统能力抽象成稳定接口，供 App 层或其他模块跨进程调用。

**分析过程：**
在智能音箱中，AIDL 适合做：

- 系统设置能力封装
- 设备信息访问
- 蓝牙/网络/音量控制
- Launcher 与系统服务通信

这样可以降低耦合，提升稳定性。

---

### Q38：AI 拾音链路你会怎么理解？
**答：**
AI 拾音不仅是“录到声音”，而是包含：

- 远场语音采集
- 麦克风阵列处理
- 波束成形
- 回声消除
- 降噪
- 增益控制
- 语音增强
- 唤醒后采样与连续识别

**分析过程：**
如果被追问，你可以说：
> 我更关注的是拾音链路的整体体验，比如语音唤醒率、识别准确率、播放场景下的回声抑制，以及连续对话的稳定性。

# 10. 高频追问与应对思路

## 10.1 “你为什么从 Java 转 Kotlin / Compose？”
回答思路：
- Kotlin 更适合现代 Android 开发
- 协程和 Flow 更适合异步和状态管理
- Compose 更适合复杂页面和声明式 UI

## 10.2 “你在鸿蒙上和 Android 的差异怎么处理？”
回答思路：
- 架构层统一抽象
- UI 层按平台实现
- 状态管理和业务逻辑尽量复用
- 平台差异通过适配层隔离

## 10.3 “音视频项目你最有把握的部分是什么？”
回答思路：
- 信令和媒体链路拆分
- 弱网和延迟优化
- 编解码适配
- 花屏、卡顿、延迟排查

## 10.4 “智能音箱项目你做得最深的点是什么？”
回答思路：
- 系统级设置与 Launcher
- AIDL 跨进程通信
- 语音链路和 AI 拾音
- 启动、ANR、Crash 稳定性治理
